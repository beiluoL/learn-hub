# 大模型应用评估：如何衡量效果

"改了 Prompt 之后到底变好还是变坏？" 没有评估体系，一切优化都是拍脑袋。

## 一、评估的三个层次

| 层次 | 方法 | 适用 |
| --- | --- | --- |
| 确定性指标 | 精确匹配、正则、JSON 校验 | 有标准答案的任务 |
| 语义指标 | 向量相似度、ROUGE/BLEU | 摘要、翻译 |
| LLM 裁判 | 用强模型给答案打分 | 开放式生成 |

## 二、RAG 专用指标

- **忠实度（Faithfulness）**：回答是否忠于检索到的上下文，不编造。
- **答案相关性（Answer Relevancy）**：回答是否切题。
- **上下文召回（Context Recall）**：检索是否覆盖了答案所需信息。

## 三、LLM-as-judge

用一个强模型按明确评分标准（rubric）给输出打分，成本低、可扩展：

```python
JUDGE_PROMPT = """
请根据以下标准给回答打分（1-5）：
- 准确性：是否与参考答案一致
- 完整性：是否覆盖要点
问题：{question}
参考答案：{reference}
待评回答：{answer}
只输出 JSON：{{"score": <int>, "reason": "<简述>"}}
"""
```

注意：裁判模型也会有偏差（如偏爱更长的回答），需人工校准。

## 四、建立评估闭环

1. 沉淀一份**黄金测试集**（覆盖典型 + 边界 case）。
2. 每次改动都跑全量评估，对比分数。
3. 线上采样 + 人工抽检，回流到测试集。

> 没有度量就没有优化。先搭最小可用的评估集，再谈提示词/检索/微调的迭代。
