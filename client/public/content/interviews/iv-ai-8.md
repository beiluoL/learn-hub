RAG 和微调不是非此即彼的替代关系，而是解决不同维度的问题。简单说: RAG 解决"知识从哪里来"（外挂记忆），微调解决"模型怎么做"（行为内化）。在实际生产系统中，两者的混合方案往往是最优解。

## RAG：给模型外挂一本参考书

### 核心优势

**知识实时更新**: 更新向量数据库即可，无需重训模型。法律条文修订、股价变动、新闻事件——数据变了，检索结果跟着变。

**幻觉可控**: 每个回答都标注了来源引用（citation），用户可以溯源验证。”模型说 X，来源是文档 Y 的第 Z 段“——这在金融/医疗/法律等强监管场景中是不可或缺的。

**低成本**: 不需要 GPU 集群做训练，只需构建检索 Pipeline + 开源 Embedding model。对于中小团队，从零到上线一个 RAG 系统的成本可以是微调方案的 1/10。

**可解释性**: 当回答出错时，可以定位到是检索模块没找到还是 LLM 理解错误。微调的"黑盒"特性让故障定位极其困难。

### 核心劣势

**检索质量是瓶颈**: RAG 是 Garbage In Garbage Out 的系统。检索不准确则 LLM 收到错误上下文，生成就错。复杂的多跳推理（"比较 A 公司的财报和 B 公司的财报中研发投入的差异"）需要检索多次，每次都可能引入误差。

**上下文窗口限制**: 即使 128K 窗口的模型，塞入过多文档会导致注意力稀释——模型对中间部分的关注度远低于开头和结尾（Lost in the Middle 效应）。

**延迟增加**: 每次请求要经过 Query 改写→向量检索→重排序→LLM 生成的多层 pipeline，端到端延迟通常是纯 LLM 调用的 2-5 倍。

**领域知识碎片化**: 如果知识散落在数百个 chunks 中，LLM 需要"拼接"这些碎片才能形成完整认知，对推理能力要求极高。

## 微调：把知识写入模型的神经元

### 核心优势

**知识内化**: 微调后的模型"真正理解了"领域概念，不需要每次检索。例如微调后的法律模型面对"合同违约条款"不需要去翻法条就能直接给出专家级回答。

**推理速度**: 不依赖外部检索的额外延迟，单次 LLM 调用即可完成。对实时性要求高的场景（客服机器人 <500ms 响应）这是决定性的。

**风格/行为定制**: 微调可以改变模型的行为模式——输出格式、语气、安全边界。RAG 只能通过 Prompt 引导，效果远不如微调稳定。

**处理隐性知识**: 有些知识难以用文本表述（写作风格/代码规范/公司内部惯例），但可以通过微调传递。RAG 对这种 Tacit Knowledge 无能为力。

### 核心劣势

**成本高**: 全参数微调（Full Fine-tuning）7B 模型就需要 4 张 A100（80G）；数据标注成本——至少需要几千条高质量 QA pairs。LoRA 虽然降低了算力需求，但数据成本仍在。

**灾难性遗忘（Catastrophic Forgetting）**: 模型在学习新领域知识时，原有通用能力可能退化。需要混合通用数据和领域数据训练，精心调参。

**知识时效性差**: 知识更新意味着重新微调。模型训练→评估→部署的周期少则数天，多则数周。对于快速变化的领域（如金融、新闻），这个延迟是不可接受的。

**维护成本**: 微调后的模型是一个"新物种"，所有监控、Prompt、评测体系都要做适配。升级基座模型时需要重新微调和评估。

## LoRA 原理：低成本微调的突破口

LoRA（Low-Rank Adaptation）的核心思想: 冻结原始模型的权重，在特定层注入可训练的**低秩矩阵**:

```
h = W * x             # 原始前向传播（W 冻结）
h = W * x + B * A * x  # LoRA 注入（B 和 A 可训练）

# W: (d x d) 原始权重，参数量 d²
# B: (d x r), A: (r x d)，参数量 2*d*r
# 当 r << d 时，可训参数极少
```

```python
import torch
import torch.nn as nn

class LoRALayer(nn.Module):
    def __init__(self, in_features, out_features, rank=8, alpha=16):
        super().__init__()
        # 原始权重，冻结
        self.linear = nn.Linear(in_features, out_features, bias=False)
        self.linear.weight.requires_grad = False
        # LoRA 低秩矩阵
        self.lora_A = nn.Parameter(torch.randn(rank, in_features) * 0.01)
        self.lora_B = nn.Parameter(torch.zeros(out_features, rank))
        self.scaling = alpha / rank

    def forward(self, x):
        return self.linear(x) + self.scaling * (x @ self.lora_A.T @ self.lora_B.T)
```

**关键参数**:
- `rank`: 低秩维度，越大容量越高但训练越慢。常见取值 4-64。
- `alpha`: 缩放因子，控制 LoRA 更新的幅度。通常设为 rank 的 1-2 倍。
- `target_modules`: 注入哪些层的 Linear/Attention 权重。常用 q_proj / v_proj。

**LoRA 的实际优势**:
- 8B 模型全参数量 8B，LoRA（r=8）仅需约 0.1% 的参数量。
- 训练速度提升 2-3 倍，显存需求降至原来的 1/3。
- 可以保存多个 LoRA Adapter（法律/医疗/金融），运行时动态切换。

## 选型决策框架

| 维度 | 选 RAG | 选微调 |
|---|---|---|
| 知识更新频率 | 高频（天/小时级） | 低频（月/季度级） |
| 数据规模 | 大（>10万文档） | 中小（几千条高质量数据） |
| 延迟要求 | 宽松（>2秒） | 严格（<500ms） |
| 预算 | 低 | 高（GPU + 标注） |
| 可解释性要求 | 高（需要来源引用） | 中低 |
| 领域特殊性 | 通用知识 | 高度专业化（医疗/法律） |
| 行为定制需求 | 弱 | 强（特定风格/格式） |
| 安全合规 | 可追溯引用 | 行为内化，难审计 |

## 混合方案：RAG + Fine-tune

业界最先进的方案是两者结合:

```
用户 Query → 微调后的模型改写 Query（更准确）
           → 检索增强（获取最新知识）
           → 微调后的模型生成回答（格式/风格一致，理解更深）
```

**典型架构**:

1. 用微调模型做 Query 理解/改写（领域 query 比通用 LLM 准 30%+）。
2. RAG 检索外部知识（保持时效性）。
3. 微调模型基于检索结果生成（风格一致、格式规范、更好地利用上下文）。

这种混合方案在金融研报生成、合同审查等场景中已验证能把准确率从纯 RAG 的 70% 提升到 90%+。

## 面试追问

- **"LoRA rank 越大越好吗？"** 不是。rank 增大带来的是容量提升但收益递减。实验表明 r=8 在很多任务上已接近 full fine-tuning 效果，r=64 通常饱和。更大的 rank 还增加了过拟合风险。
- **"什么时候必须用全量微调而不是 LoRA？"** 当需要学习全新的 token（扩展词表）、需要大幅改变模型行为（从一个英文作家变成一个中文程序员）、或者 LoRA 效果显著低于 full fine-tuning 时。对于绝大多数领域适配，LoRA 优先。
- **"怎么检测灾难性遗忘？"** 在微调前后分别用通用 benchmark（MMLU/HellaSwag）测试，看性能退化幅度。训练 loss 曲线如果验证集 loss 先降后升就是明显的信号。
- **"用 RAG 的方案，Embedding 模型需要微调吗？"** 如果处于高度专业化领域（医学术语/法律条文），通用 Embedding 模型可能表现不佳，此时微调 Embedding 模型（如用 Sentence Transformers）可以显著提升检索 recall。
