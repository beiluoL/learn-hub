技术选型的本质不是"哪个技术更好"，而是**在当前约束下哪个技术最合适**。优秀的架构师从不依赖个人偏好做决策，而是用一个结构化的框架来衡量——业务需求、团队能力、社区生态、长期维护四个维度，每个维度必须加权且有理有据。盲目追新和不考虑团队现状是两种最典型也最致命的错误。

## 决策框架：四维评分模型

### 维度 1：业务需求匹配度（权重 35%）

技术必须服务于业务，而不是反过来:

- 这个技术的核心设计是否契合我们的业务场景？强一致性的金融系统不应该用最终一致的存储方案。
- 性能要求: 延迟 <50ms？QPS >10000？数据量 TB 级？
- 功能要求: 需要全文搜索 → ES；需要地理空间查询 → PostgreSQL + PostGIS。
- 合规要求: 必须私有部署？数据不出境？

### 维度 2：团队能力匹配度（权重 30%）

一个完美的技术如果团队没人会用，就是最差的选择:

- 团队当前的技术栈是什么？一个 Java 团队突然引入 Go 服务会带来陡峭的学习曲线。
- 学习成本: 新手多久能独立开发？是否有培训资源？
- 是否有核心人员能兜底？1 个专家带 4 个新手 > 5 个新手自己摸索。

### 维度 3：社区生态健康度（权重 20%）

- GitHub Stars / 活跃 Issue 数 / PR 合并速度。
- 插件/库丰富程度: 一个 ORM 框架是否支持你用的数据库？
- Stack Overflow 问答数量: 遇到问题时能搜到的答案多不多？
- 是否有商业公司支持？如 Elastic 之于 Elasticsearch，HashiCorp 之于 Terraform。

### 维度 4：长期可维护性（权重 15%）

- 版本迭代节奏: 过于激进（周更 breaking change）或过于缓慢（年更）都不好。
- 向后兼容性承诺: 升级大版本是否必须大量改代码？
- 核心维护者数量: bus factor=1（只有 1 个核心维护者）是巨大风险。
- 许可证风险: AGPL 对企业内部使用不友好。

## 案例一：React vs Vue

### 决策表

| 维度 | React | Vue | 本团队权重 |
|---|---|---|---|
| 学习曲线 | 中（需理解 Hooks/JSX/闭包陷阱） | 低（模板语法直观） | 团队 3 个后端转前端，Vue 胜 |
| TypeScript 支持 | 优秀（天然配合 TS） | 良好（3.x 改进大） | TS 强制用，React 略优 |
| 生态系统 | 极大（Next.js/React Native） | 大（Nuxt/Element Plus） | 需要 SSR + 移动端 → React |
| 市场招聘 | 岗位多，人才池大 | 国内中小企业为主 | 长期招人难易度 → React |
| 升级成本 | 需关注（18→19 有变化） | 平滑（2→3 重大，3 以后平滑） | Vue 2→3 的迁移教训 |
| 性能 | 同级 | 同级 | 平手 |

### 决策结论

- **选 React**: 如果团队需要做 SSR（Next.js）、React Native 移动端、或者在一线城市招聘。
- **选 Vue**: 如果团队前端人员少、来自后端背景、项目是中小型管理系统、模板驱动的 UI 多于交互密集型 UI。
- **不要选**: 两者混用。除非有非常明确的网关隔离（不同子应用独立技术栈的微前端架构），否则只会增加认知负担。

## 案例二：MySQL vs PostgreSQL

### 决策表

| 维度 | MySQL | PostgreSQL | 场景 |
|---|---|---|---|
| 事务支持 | 默认 RR 隔离，无 SERIALIZABLE | 完整 ACID，真正的 SERIALIZABLE（SSI） | 金融交易 → PG |
| JSON 支持 | 5.7+ 有 JSON 类型，函数简单 | 原生 JSONB + GIN 索引，JSONPath 查询 | 灵活 Schema → PG |
| 扩展性 | 插件生态弱 | 强大（PostGIS 地理/Citus 分布式/Vector 向量） | 多场景 → PG |
| 全文搜索 | 内置中文分词弱 | 内置 TSVector + 可扩展开源分词 | 轻量搜索 → PG |
| 复制与高可用 | MGR/Orchestrator 成熟 | Streaming Replication + Patroni 成熟 | 平手 |
| 运维成本 | 低（DBA 多，文档多） | 中（Vacuum 调优需要经验） | 大厂有 DBA → 平手 |
| 社区成熟度 | 极高（互联网标配） | 高（增速最快） | 平手 |

### 决策结论

- **选 MySQL**: 团队有 MySQL DBA / 纯 CRUD 业务 / 互联网公司标准栈。
- **选 PostgreSQL**: 需要复杂查询（窗口函数/CTE）、地理空间、JSON 文档存储、向量搜索（pgvector）、或对数据一致性有极高要求。
- **新项目默认推荐 PostgreSQL**（2024 年后），除非有明确的 MySQL 依赖（如某些 SaaS 只支持 MySQL）。

## 案例三：Kafka vs RabbitMQ vs RocketMQ

| 维度 | Kafka | RabbitMQ | RocketMQ | 场景 |
|---|---|---|---|---|
| 吞吐量 | 百万级/秒 | 万级/秒 | 十万级/秒 | 高吞吐 → Kafka |
| 延迟 | P99 ~10ms | P99 ~1ms | P99 ~1ms | 低延迟 → RabbitMQ |
| 消息可靠性 | At-Least-Once | At-Most-Once(默认) / 手动 ACK | At-Least-Once | 可靠 → Kafka/RocketMQ |
| 事务消息 | 支持（事务 API） | 不支持（需自己实现） | 原生支持 | 分布式事务 → RocketMQ |
| 消息顺序 | 分区内有序 | 队列内有序（单消费者） | 队列内有序 | 顺序消息 → 都可 |
| 延时消息 | 不支持原生 | 插件支持 | 原生支持（18 级） | 延时 → RocketMQ |
| 运维复杂度 | 中（依赖 ZooKeeper/KRaft） | 低 | 低（Java 技术栈友好） | Java 团队 → RocketMQ |
| 生态 | 最丰富 | 丰富（AMQP 标准） | 阿里系 | 通用 → Kafka |

### 决策结论

- **选 Kafka**: 大数据/流处理/日志采集/事件溯源架构。几乎所有中大型互联网公司的默认选择。
- **选 RabbitMQ**: 业务系统中低吞吐量的异步任务（发邮件/短信），对低延迟有要求。
- **选 RocketMQ**: Java 团队 + 需要分布式事务消息（电商订单场景），阿里系技术栈。

## 选型流程的实践步骤

```
1. 明确需求（写下来，不是脑子里想）
   ↓
2. 列出候选方案（2-3 个，不要 5+ 个）
   ↓
3. 技术调研（每人负责一个候选方案，调研 2-3 天）
   ↓
4. PoC（概念验证）——用最简单的方式跑通核心场景
   ↓
5. 写出 ADR（Architecture Decision Record）文档
   ↓
6. 团队评审 + 最终决策
```

### ADR 文档模板

```markdown
# ADR-001: 选择 PostgreSQL 作为主数据库

## 状态
已接受（2024-01-15）

## 背景
新项目需要支持复杂的地理空间查询和 JSON 灵活 Schema。

## 决策
使用 PostgreSQL 15+ 替代 MySQL 8.0。

## 方案对比
- PostgreSQL: 原生 PostGIS 地理查询，JSONB 索引，CTE 复杂查询。
- MySQL 8.0: 地理功能弱，JSON 函数少，缺少部分窗口函数。
- MongoDB: 文档模型很适合但缺少事务 JOIN，多表关联查询体验差。

## 后果
- 正面: 一套数据库满足地理 + JSON + 关系查询，减少技术栈。
- 负面: 团队无 PG DBA，需 2 周学习 + 搭建监控。
- 风险: Vacuum 调优需要持续投入，已联系外部顾问支持。
```

## 常见错误与避免之道

| 错误 | 为什么错 | 正确做法 |
|---|---|---|
| 盲目追新（用最新版） | 文档少/社区小/未知 bug | 选择已经过 2-3 年生产验证的版本 |
| 不考虑团队能力 | 技术很好但团队跟不上 | 团队能力占 30% 权重，不要忽视 |
| 过度设计 | 日活 100 用户上了 K8s+Istio | 从简单方案起步，有需求再演进 |
| 迷信文档 | 官方文档总说"完美支持" | 一定要做 PoC，在生产量级上压测 |
| 不考虑可替代性 | 单一供应商锁定 | 评估迁移成本，给未来留后路 |
| "大厂用这个所以我们也用" | 大厂有专门的团队维护 | 评估自己的团队规模和运维能力 |

## 面试追问

- **"如果团队坚决反对你的技术选型怎么办？"** 首先确认团队反对的是技术本身还是变化带来的不确定性。如果是前者，用 PoC 数据说话。如果是后者，做充分的知识分享和培训计划，降低焦虑。但绝对不能用"我是架构师听我的"这种态度——一个靠权威推行的方案，即使技术上正确，也会因为团队抵触而失败。
- **"你选了一个新技术，上线后发现问题不断，怎么办？"** 立即评估是"能被修复的问题"还是"根本性不适合"。前者继续投入人力修复，后者要有壮士断腕的勇气——立刻启动回退到备用方案。选型时就要准备 plan B，这是架构师的基本素养。
- **"选开源还是自研？"** 99% 情况下选开源。自研只有在以下条件同时满足时考虑: 开源方案有无法修复的致命缺陷 + 团队有核心研发能力 + 业务足够特殊形成了护城河。否则自研只会产出一堆低质量且无人维护的内部系统。
