## TCP 三次握手与四次挥手

**核心结论**：TCP 三次握手用于建立可靠双向连接，四次挥手用于安全断开连接。握手必须三次的核心原因是**防止已失效的连接请求报文段突然又传到服务端而产生错误**。

---

### 三次握手过程（文字时序）

```
客户端                              服务端
CLOSED                              LISTEN
  |                                    |
  |------SYN (seq=x)----------------->>| SYN_SENT
  |                                    | SYN_RCVD
  |<<----SYN+ACK (seq=y, ack=x+1)----|
  |                                    |
  |------ACK (ack=y+1)-------------->>| ESTABLISHED
  |                                    |
ESTABLISHED                            |
```

**第一步**：客户端发送 SYN 报文，携带初始序列号 x，客户端进入 SYN_SENT 状态。

**第二步**：服务端收到 SYN 后，回复 SYN+ACK 报文，携带自己的序列号 y 和对客户端序列号的确认 ack=x+1，服务端进入 SYN_RCVD 状态。

**第三步**：客户端收到 SYN+ACK 后，回复 ACK 确认报文 ack=y+1，双方进入 ESTABLISHED 状态，连接建立完成。

---

### 为什么是三次而不是两次？

**核心原因**：防止历史连接。

考虑如下场景：客户端发送的第一个 SYN 在网络中滞留。客户端超时后重发 SYN 并成功建立连接、传输数据、关闭连接。此时，之前滞留的 SYN 到达服务端。

- **两次握手**：服务端收到滞留 SYN 后立即进入 ESTABLISHED，并为该连接分配资源，回复 SYN+ACK。但客户端此时已经关闭，不会回复，服务端将一直等待，造成资源浪费。
- **三次握手**：服务端收到滞留 SYN 后进入 SYN_RCVD 并回复 SYN+ACK。客户端收到后发现这不是自己发起的连接请求，发送 RST 复位报文终止该连接。服务端不会错误地建立连接。

从信息论角度：三次是双方确认自己发送和接收能力正常的最小交互次数——客户端需要确认自己能发能收，服务端也需要确认自己能发能收。

**为什么不是四次**：第二步（SYN+ACK）可以合并发送，不需要分成两步，合并后恰好三次。四次没有额外收益。

---

### 四次挥手过程（文字时序）

```
客户端                              服务端
ESTABLISHED                         ESTABLISHED
  |                                    |
  |------FIN (seq=u)--------------->>| FIN_WAIT1
  |                                    | CLOSE_WAIT
  |<<----ACK (ack=u+1)-------------|
  |FIN_WAIT2                           |
  |                                    | (服务端处理剩余数据)
  |<<----FIN (seq=v)---------------| LAST_ACK
  |TIME_WAIT                           |
  |------ACK (ack=v+1)------------->>| CLOSED
  | (等待 2MSL)                        |
CLOSED                                  |
```

**第一步**：主动关闭方（客户端）发送 FIN 报文，进入 FIN_WAIT1 状态。

**第二步**：被动关闭方（服务端）收到 FIN 后回复 ACK，进入 CLOSE_WAIT 状态。客户端收到 ACK 后进入 FIN_WAIT2。此时 TCP 处于半关闭状态——客户端不再发送数据，但服务端还可以发送。

**第三步**：服务端处理完剩余数据后，发送 FIN 报文，进入 LAST_ACK 状态。

**第四步**：客户端收到 FIN 后回复 ACK，进入 TIME_WAIT 状态，等待 2MSL 后关闭。服务端收到 ACK 后立即关闭。

---

### TIME_WAIT 为什么等 2MSL

MSL（Maximum Segment Lifetime）是报文最大生存时间，通常为 30 秒~2 分钟。

**两个原因**：

1. **确保最后一个 ACK 能被对方收到**：如果 ACK 丢失，服务端会重发 FIN，客户端有足够时间重新发送 ACK。
2. **让本次连接内所有报文从网络中消失**：防止旧连接的延迟报文串扰到使用相同四元组的新连接。

等待时间为 2MSL = 1MSL（ACK 丢失的最大时间）+ 1MSL（FIN 重传的最大到达时间）。

---

### 大量 TIME_WAIT 的解决

**场景**：高并发短连接（如 HTTP 短连接）会产生大量 TIME_WAIT，导致端口耗尽（Linux 默认约 28000 个可分配端口）。

**解决方案**：

1. **开启 tcp_tw_reuse**（内核 >= 2.6.26）：

```bash
echo 1 > /proc/sys/net/ipv4/tcp_tw_reuse
```

允许将 TIME_WAIT 状态的 socket 重新用于新的 TCP 连接，仅对客户端有效。

2. **启用快速回收**（谨慎使用，NAT 环境不推荐）：

```bash
echo 1 > /proc/sys/net/ipv4/tcp_tw_recycle
```

3. **使用连接池**：客户端复用已有连接，避免频繁新建。

4. **长连接**：HTTP/1.1 Keep-Alive 头部 + 连接池，减少连接创建和关闭。

5. **调整系统参数**：

```bash
# 增大端口范围
echo "10000 65000" > /proc/sys/net/ipv4/ip_local_port_range
# 加速 FIN_WAIT2 回收
echo 30 > /proc/sys/net/ipv4/tcp_fin_timeout
```

6. **应用层**：让服务端主动关闭连接，TIME_WAIT 会落在服务端而非客户端。

### 面试官追问

**追问**：如果第三次握手 ACK 丢失了怎么办？

**回答**：客户端进入 ESTABLISHED（认为连接已建立），服务端在 SYN_RCVD 等待。服务端没有收到 ACK 会超时重传 SYN+ACK，重传次数由 tcp_synack_retries 控制（默认 5 次）。如果仍无回应，服务端关闭半连接。客户端此时发送数据，服务端收到后会回复 RST，客户端的 write 系统调用返回 ECONNRESET。

**追问**：SYN Flood 攻击如何防御？

**回答**：攻击者发送大量 SYN 但不回复 ACK，填满服务端 SYN 半连接队列。防御手段：SYN Cookie（利用 hash 计算序列号避免在 SYN 阶段分配资源）、增大半连接队列、缩短 SYN 超时时间、防火墙限流。

**追问**：四次挥手可以三次吗？

**回答**：当被动关闭方没有数据要发送时，可以将第二步的 ACK 和第三步的 FIN 合并为一个报文，形成"三次挥手"——但这依赖时机，标准流程仍是四次。
