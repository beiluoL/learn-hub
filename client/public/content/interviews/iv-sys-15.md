## HTTPS 握手与加密原理

**核心结论**：HTTPS 采用**混合加密**——非对称加密（RSA/ECDHE）安全交换对称密钥，对称加密（AES）高效传输数据。两者结合兼顾安全性与性能。

---

### 对称加密 vs 非对称加密

**对称加密**：加密与解密使用**同一把密钥**。

| 特点 | 说明 |
|------|------|
| 密钥 | 一把密钥，双方共享 |
| 速度 | 快（CPU 友好，通常有硬件加速） |
| 典型算法 | AES、ChaCha20、DES、3DES |
| 主要问题 | **密钥分发**——如何安全地将密钥传给对方 |
| 适用场景 | 大数据量加密传输 |

**非对称加密**：使用**一对密钥**（公钥 + 私钥）。公钥加密的数据只有私钥能解密，私钥签名的数据可用公钥验签。

| 特点 | 说明 |
|------|------|
| 密钥 | 公钥（公开）+ 私钥（保密） |
| 速度 | 慢（比对称加密慢 100~1000 倍） |
| 典型算法 | RSA、ECC、ECDHE |
| 主要优势 | **解决密钥分发**——公钥可以公开传输 |
| 适用场景 | 数字签名、密钥交换 |

**为什么不用纯非对称加密**：太慢。非对称加密涉及大数模幂运算，RSA 2048 位加密比 AES-128 慢约 500 倍，不适合加密大量数据。

**为什么不用纯对称加密**：密钥分发是死循环——要传输对称密钥，又需要安全信道；要建立安全信道，又需要对称密钥。非对称加密解决了这个鸡生蛋问题。

### TLS 1.2 握手过程（文字时序）

```
客户端                                    服务端
  |                                         |
  |--ClientHello (支持版本、加密套件、随机数1)-->>|
  |                                         |
  |<<--ServerHello (选定的版本/套件、随机数2)-----|
  |<<--Certificate (服务端证书)---------------|
  |<<--ServerHelloDone----------------------|
  |                                         |
  | (验证证书：颁发机构/有效期/域名匹配)           |
  | (生成 Pre-Master Secret)                 |
  |--ClientKeyExchange (RSA加密的Pre-Master)-->>|
  |--ChangeCipherSpec (后续用加密)----------->>|
  |--Finished (加密的握手摘要)--------------->>|
  |                                         |
  |                            (解密Pre-Master)|
  |                            (生成会话密钥)   |
  |<<--ChangeCipherSpec (后续用加密)----------|
  |<<--Finished (加密的握手摘要)--------------|
  |                                         |
  |============ 安全通信开始 =================|
```

**详细步骤**：

1. **ClientHello**：客户端发送支持的 TLS 版本、加密套件列表（如 TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256）、客户端随机数、Session ID（支持会话恢复时使用）。

2. **ServerHello**：服务端选定 TLS 版本和加密套件，发送服务端随机数。

3. **Certificate**：服务端发送证书链（包含公钥）。

4. **ServerHelloDone**：服务端表示本轮消息结束。

5. **客户端验证证书**：
   - 验证证书链：用操作系统/浏览器的内置根证书逐级验证签发方签名，直到找到可信根。
   - 验证有效期。
   - 验证域名匹配（SAN 或 CN 字段）。
   - 检查证书吊销状态（OCSP 或 CRL）。

6. **密钥交换（以 RSA 为例）**：客户端生成 48 字节的 Pre-Master Secret，用服务端公钥加密后通过 ClientKeyExchange 发送。

7. **生成会话密钥**：双方均用客户端随机数 + 服务端随机数 + Pre-Master Secret，通过 PRF（伪随机函数）推导出一组对称密钥（加密密钥、MAC 密钥、IV）。

8. **ChangeCipherSpec + Finished**：双方通知对方后续通信加密，发送 Finished 消息（整个握手过程的摘要加密），用于验证握手未被篡改。

### TLS 1.3 的优化

TLS 1.3 将握手优化到 **1-RTT**，主要改进：

1. **密钥交换前置**：ClientHello 直接带上密钥协商参数（支持 ECDHE 的 Key Share），不需要等 ServerHello 后再发送。
2. **只支持前向安全的算法**：去掉了不支持 PFS 的 RSA 密钥交换和 CBC 模式加密。
3. **缩减加密套件**：不再发送完整套件，只发送几个关键参数（AEAD 算法名 + 哈希名）。
4. **0-RTT 模式**：对于已连接过的客户端，可以直接发送应用数据（带重放风险警告）。

### 证书链验证

```
根 CA 证书（操作系统内置信任）
  └── 中间 CA 证书（由根 CA 签发）
        └── 服务端证书（由中间 CA 签发，包含域名 www.example.com）
```

客户端启动时预装了约 150 个根 CA 证书。验证过程：
1. 检查服务端证书的签发者，找到对应中间 CA 证书。
2. 用中间 CA 的公钥验证服务端证书的数字签名。
3. 检查中间 CA 证书的签发者，找到根 CA。
4. 用根 CA 公钥验证中间 CA 的数字签名。
5. 根 CA 自签名证书直接信任。

### 面试官追问

**追问**：中间人攻击（MITM）为什么对 HTTPS 无效？

**回答**：攻击者在客户端和服务端之间转发通信。如果攻击者伪造服务端证书发给客户端，客户端验证时会发现签发者不在信任列表或域名不匹配，浏览器会显示证书警告。除非用户主动安装了恶意根证书，否则无法实施 MITM。

**追问**：前向安全性（Forward Secrecy）是什么？

**回答**：即使服务端私钥未来泄露，历史通信也无法被解密。RSA 密钥交换不具备前向安全性（私钥解密 Pre-Master → 推导会话密钥 → 解密历史），而 ECDHE 每次会话生成临时的公私钥对，私钥仅用于签名而非加密传输密钥。TLS 1.3 强制使用 ECDHE，所有通信都具备前向安全性。

**追问**：HTTPS 降级攻击怎么防御？

**回答**：攻击者修改 ClientHello 去掉高版本 TLS，迫使使用存在已知漏洞的旧版本。防御手段：服务端强制启用 HSTS（HTTP Strict Transport Security），浏览器记住该域名只允许 HTTPS；TLS_FALLBACK_SCSV 伪套件机制检测降级。
