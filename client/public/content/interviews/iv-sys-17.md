## 进程、线程与协程

**核心结论**：进程是资源分配的基本单位，线程是 CPU 调度的基本单位，协程是用户态的轻量级并发单元。从进程到协程，调度层级逐级上移，上下文切换开销逐级降低。

---

### 进程 vs 线程

**进程**：运行中的程序实例，拥有独立的地址空间（代码段、数据段、堆、栈）、文件描述符表、信号处理表等系统资源。

**线程**：进程内的执行流，共享进程的地址空间和文件描述符表，但拥有独立的栈、寄存器和线程局部存储（TLS）。

| 维度 | 进程 | 线程 |
|------|------|------|
| 资源拥有 | 独立资源（内存/文件/信号） | 共享父进程资源 |
| 地址空间 | 独立（每个进程独立页表） | 共享（同一页表） |
| 通信方式 | IPC（管道/消息队列/共享内存/Socket） | 直接读写共享内存 |
| 创建开销 | 大（fork + exec，复制页表） | 小（分配栈 + TCB） |
| 上下文切换 | 大（刷新 TLB，切换页表） | 中（切换寄存器 + 内核栈） |
| 隔离性 | 强（一个进程崩溃不影响其他） | 弱（一个线程崩溃可能带崩整个进程） |
| 适用场景 | 高可靠性、多核心利用 | 高并发、IO 密集型 |

---

### 上下文切换开销对比

**进程切换**：需要切换页表基址寄存器（CR3 on x86），触发 TLB 全刷新（Flush），后续内存访问需要重新进行页表遍历，代价最高，通常在微秒级（1~3us）。

**线程切换**：同一进程内线程切换不需要切换页表，只需保存和恢复寄存器（PC、SP、通用寄存器等）、切换内核栈。TLB 仍然有效，延迟大幅降低。

**协程切换**：发生在用户态，不涉及内核。只需保存少量寄存器（栈指针、帧指针）并交换协程的栈空间指针，通常在纳秒级别（几十~几百 ns）。

```
开销对比（数量级）：
进程切换  >  线程切换  >  协程切换
 1-3us      ~1us       ~10-100ns
```

### 协程的优势

**协程**（Coroutine）是完全由用户态程序控制的并发单元。调度权从操作系统内核"上移"到应用程序自身。

**核心优势**：

1. **极低的切换开销**：不经过内核态，不需要系统调用，仅仅是一个函数栈的保存和恢复。Golang 的 Goroutine 切换成本约几十纳秒，一个线程切换成本是其百倍以上。

2. **同步风格写异步代码**：协程通过挂起（yield/suspend）和恢复（resume）机制，让开发者以同步的、直觉化的方式编写异步逻辑，消除回调地狱。

```javascript
// 协程风格：像同步代码一样直观
async function fetchUserData(userId) {
    const user = await db.query(`SELECT * FROM users WHERE id = ${userId}`);
    const orders = await db.query(`SELECT * FROM orders WHERE user_id = ${userId}`);
    return { user, orders };
}
// 如果不使用协程（传统回调方式）：
// db.query(sql1, (err, user) => {
//     db.query(sql2, (err, orders) => {
//         // 两层嵌套，复杂逻辑下形成"回调地狱"
//     });
// });
```

3. **高并发能力**：线程受限于内核调度和内存开销（Linux 默认线程栈 8MB），一个进程很难创建超过几千个线程。而协程的栈是动态、可增长的，一个 Goroutine 初始仅占用约 2KB，Golang 程序可以轻松跑百万 Goroutine。

4. **无锁同步**：协程的调度由应用控制，可以在关键区域不主动挂起，相当于天然的单线程运行，避免多线程竞争和加锁成本。

**典型实现对比**：

| 语言/框架 | 协程机制 | 特点 |
|-----------|----------|------|
| Golang | Goroutine + Channel | M:N 调度（多协程映射到多线程），Work Stealing |
| Python | async/await (asyncio) | 单线程事件循环，需显式 await |
| JavaScript | async/await (Promise) | 事件循环 queue，单线程 |
| Kotlin | suspend 函数 | 编译器生成状态机，编译期转换 |
| C++20 | co_await/co_yield | 无栈协程，编译器实现 |

### CPU 密集 vs IO 密集选型

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| CPU 密集型（计算密集） | 多进程 / 线程池（线程数 ≈ CPU 核心数） | 计算占满 CPU，增加线程只会增多上下文切换 |
| IO 密集型 | 协程 / 异步 IO | CPU 大部分时间在等待，协程可以在等待时让出 CPU 给其他协程 |
| 混合型 | 协程 + 小线程池 | 真正计算的部分用线程池避免阻塞调度器 |

### 面试官追问

**追问**：Goroutine 的 M:N 调度模型是什么？

**回答**：Golang 使用 GMP 模型——G（Goroutine，协程）、M（Machine，操作系统线程）、P（Processor，逻辑处理器）。P 的数量由 GOMAXPROCS 决定（默认 CPU 核心数）。每个 P 维护一个本地 G 队列，M 必须绑定 P 才能执行 G。当 M 因系统调用阻塞时，P 会与新的 M 绑定继续执行，原 M 等待系统调用完成。Work Stealing 机制让空闲的 P 可以从其他 P 的队列中"偷" Goroutine 执行，保证负载均衡。

**追问**：多线程的多核并行与协程的单线程并发有何本质区别？

**回答**：并行是同时执行多个任务（物理上的同时），并发是交替执行多个任务（逻辑上的同时）。多线程在多核机器上可以实现真正的并行——不同核心同时执行不同线程。单线程协程本质是并发——在多个任务之间快速交替，同一时刻只有一个任务在执行。但协程也可以配合多线程实现并行+并发的组合。
