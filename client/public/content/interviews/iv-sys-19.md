## 虚拟内存与分页分段

**核心结论**：虚拟内存让每个进程拥有独立的、连续的虚拟地址空间，通过 MMU 透明地映射到物理内存——解决了地址隔离、内存超卖（换入换出）和碎片三大问题。分页和分段是两种不同的映射策略，现代操作系统以分页为主。

---

### 虚拟内存与 MMU

**虚拟地址 → 物理地址转换流程**：

```
虚拟地址（32位示例）：
| 页目录索引(10bit) | 页表索引(10bit) | 页内偏移(12bit) |
      ↓
  CR3 → 页目录表（每进程一个）
           ↓
         页表项 PTE
           ↓
      物理页框基址 + 页内偏移 → 物理地址
```

**MMU（Memory Management Unit）** 是 CPU 内部的硬件单元，负责拦截所有的虚拟地址访问并完成地址转换。转换依赖**页表**（每进程一套独立的页表，存储在操作系统的内存中）。

**TLB（Translation Lookaside Buffer）** 是 MMU 内部的高速缓存，存储最近转换过的虚拟→物理地址映射，避免每次都要走完整的页表遍历路径。TLB 命中时地址转换仅需要 1 个 CPU 时钟周期。

**虚拟内存的作用**：

1. **隔离**：每个进程有独立的虚拟地址空间，A 进程无法访问 B 进程的内存。
2. **抽象**：进程看到的是连续的、从 0 开始的地址空间，底层物理内存可以是不连续的碎片。
3. **超卖**：所有进程的虚拟地址总和可以超过物理内存总量，通过页面置换将不活跃页面换出到磁盘（swap）。
4. **共享**：同一个物理页面可以映射到多个进程的虚拟地址空间（如动态链接库 libc.so 所有进程共享一份物理页）。

### 分页 vs 分段

**分页**：将虚拟地址空间和物理内存都划分为固定大小的块（页/页框），常见大小为 4KB。不以程序的逻辑结构为单位。

**分段**：用程序的逻辑结构（代码段、数据段、栈段、堆段）为单位划分。段的大小可变。

| 维度 | 分页 | 分段 |
|------|------|------|
| 划分单位 | 固定大小（4KB/2MB/1GB） | 逻辑单元（代码/数据/栈），大小可变 |
| 碎片 | 内部碎片（最后一页未用完） | 外部碎片（多个段释放后留下不连续小空闲区） |
| 程序员可见性 | 透明（程序员无感知） | 可见（程序员可用段寄存器） |
| 共享与保护 | 困难（以页为单位，可能与逻辑边界不对齐） | 自然（每个段天然是对应的逻辑单元） |
| 地址结构 | 页号 + 页内偏移 | 段号 + 段内偏移 |
| 现代操作系统 | 主流（几乎所有 OS 采用） | 配合分页使用（Linux 分段仅限于满足 x86 遗留要求） |

**为什么分页胜出**：分页避免了外部碎片——物理内存永远是页对齐的，分配和回收都是整页操作，不需要紧凑（Compaction）。而分段需要处理变长分配，长时间运行后物理内存变成"瑞士奶酪"，必须通过紧凑整理，代价极大。

### 多级页表

现代 64 位系统使用 4 级或 5 级页表：

```
虚拟地址 64位（实际只用 48 位）：
| PGD(9bit) | PUD(9bit) | PMD(9bit) | PTE(9bit) | offset(12bit) |
```

**为什么需要多级**：以 4KB 页为例，64 位系统的一级页表条目数 = 2^52 / 12 = 2^40 个 PTE，每个 PTE 8 字节，总共 8TB——显然不可能为每个进程分配这么大的页表。多级页表只在需要时才分配中间级别的页表页，实际占用远小于理论最大值。

### 缺页中断处理流程

缺页中断（Page Fault）发生在进程访问一个虚拟地址，但 MMU 发现该页不在物理内存中时。

**处理流程**：

```
① CPU 执行访存指令（如 mov eax, [0x7f8a0000]）

② MMU 查询 TLB → 未命中
   ↓
③ 遍历页表，找到对应 PTE → PTE 的 Present 位 = 0（表示不在物理内存中）
   ↓
④ MMU 触发缺页异常（#PF, Page Fault Exception），CPU 切换到内核态
   ↓
⑤ 操作系统缺页中断处理程序（do_page_fault）开始执行：
│
├─ 情况 A：页在磁盘 swap 中（Minor Fault 之外的情况）
│  ├─ 在交换空间中查找该页面
│  ├─ 在物理内存中找一个空闲页框（若无，触发页面置换）
│  ├─ 从磁盘读入页面内容到该页框
│  ├─ 更新 PTE（填入页框地址，置 Present = 1）
│  └─ 刷新 TLB 相关条目
│
├─ 情况 B：匿名页初次访问（第一次写时分配，Copy-On-Write）
│  └─ 分配一个清零后的新物理页
│
├─ 情况 C：页在页缓存（Page Cache）中但未映射
│  └─ 直接建立映射，不需要磁盘 IO
│
└─ 情况 D：访问了非法地址（如空指针）
   └─ 向进程发送 SIGSEGV 信号（Segmentation Fault），进程终止
│
⑥ 中断返回，CPU 重新执行那条访存指令
   ↓
⑦ MMU 再次查询 TLB → 已加载 → 地址转换成功 → 正常访问数据
```

**分类**：

| 类别 | 触发条件 | 处理开销 |
|------|----------|----------|
| Minor Fault | 页在内存但未建立映射（如 COW 触发、页缓存命中） | 极小（仅 CPU 操作，无 IO） |
| Major Fault | 页不在内存，需要从磁盘读取 | 大（涉及磁盘 IO，毫秒级） |
| Invalid Fault | 地址非法（空指针、越界） | 不处理，发 SIGSEGV |

### 页面置换算法

当物理内存满时，需要选择牺牲页换出。

| 算法 | 原理 | 特点 |
|------|------|------|
| OPT（最优） | 换出未来最长时间不再访问的页 | 理论最优，实际无法实现（无法预知未来） |
| FIFO | 换出最先进入的页 | 实现简单，性能差，有 Belady 异常 |
| LRU | 换出最近最少使用的页 | 性能好，需额外硬件支持，开销大 |
| Clock（NRU） | LRU 的近似实现，用访问位 + 修改位 | Linux 实际使用，性能接近 LRU |
| LFU | 换出访问频率最低的页 | 可能锁死旧热门页 |

**Linux 的置换策略**：使用经过改进的 LRU 算法，将内存页分为 Active List（活跃链表）和 Inactive List（非活跃链表）。Page Cache 和匿名页分属不同链表，通过 kswapd 内核线程在后台执行页面回收。

### Linux 交换空间（Swap）

```bash
# 查看 swap 使用情况
swapon --show
# NAME      TYPE SIZE USED PRIO
# /swapfile file   4G  512M   -2

# 查看每个进程的 swap 占用
for file in /proc/*/status; do
    awk '/VmSwap|Name/{printf $2 " " $3 "\t"} END{print ""}' "$file"
done | grep -v "0 kB" | sort -k2 -n -r | head -10

# 调整交换倾向（默认 60，越小越倾向于使用物理内存）
echo 10 > /proc/sys/vm/swappiness
```

### 面试官追问

**追问**：为什么 32 位系统每个进程最大支持 4GB 虚拟地址空间？

**回答**：32 位地址总线宽度为 32 位，可表达的地址数量为 2^32 = 4,294,967,296，即 4GB。这是硬件寻址能力的上限。实际 32 位 Linux 默认使用 3GB/1GB 分割——高 1GB 留给内核空间，用户进程最多使用 3GB。

**追问**：什么是 TLB Shootdown？

**回答**：当操作系统修改页表后（如替换了某个页面），不同 CPU 核心上的 TLB 缓存了该过时条目。需要通过 IPI（Inter-Processor Interrupt）通知所有核心刷新对应的 TLB 条目，这个过程称为 TLB Shootdown。频繁的 TLB Shootdown 是大型多核系统性能瓶颈之一。

**追问**：大页（Huge Page）是什么优势？

**回答**：使用 2MB 或 1GB 的大页代替默认的 4KB 小页。优势：TLB 条目数量固定（通常几百条），每个大页条目覆盖更多地址空间，大幅提升 TLB 命中率；减少页表遍历层级（2MB 页从 4 级减少到 3 级）。代价：内部碎片增加（页未用完），内存申请延迟变大。数据库和 JVM 是大页的典型受益者。
