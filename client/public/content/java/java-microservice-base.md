# 微服务架构基础：注册发现、配置中心与拆分原则

当单体应用变得庞大、迭代缓慢、一处故障全盘皆崩时，微服务架构就成为常见的演进方向。本篇帮你建立微服务的核心心智模型：它解决什么问题、服务之间如何找到彼此、配置如何统一管理，以及该怎样拆分。

## 一、单体 vs 微服务

**单体架构**：所有功能打包在一个应用、一个进程里。优点是开发部署简单、事务和调用都在本地；缺点是随着代码膨胀，编译慢、启动慢、技术栈被绑定、任何小改动都要全量发布，且一个模块的内存泄漏会拖垮整个应用。

**微服务架构**：把系统按业务域拆成多个独立部署、独立运行的小服务，服务间通过网络通信。每个服务可以独立开发、部署、扩容，技术栈也可不同。

| 维度 | 单体 | 微服务 |
| --- | --- | --- |
| 部署 | 一次整体发布 | 各服务独立发布 |
| 扩展 | 整体扩容 | 按需对热点服务扩容 |
| 容错 | 一处崩全崩 | 故障隔离在单服务内 |
| 复杂度 | 代码耦合高 | 分布式复杂度高（网络、一致性） |

微服务不是银弹，它引入了分布式事务、服务治理、链路追踪等新难题，团队规模和数据一致性要求决定了是否值得拆。

## 二、服务注册与发现

在微服务中，服务实例的 IP 和端口是动态变化的（尤其容器化部署）。**服务注册发现**解决了“调用方如何找到提供方”的问题。

**角色**：
- **Provider（服务提供者）**：启动时把自己的地址、服务名注册到注册中心。
- **Consumer（服务消费者）**：从注册中心拉取可用实例列表，再发起调用。
- **Registry（注册中心）**：维护服务名到实例列表的映射，并提供心跳检测剔除宕机实例。

**基本流程**：
1. Provider 启动，向 Registry 注册元数据（服务名、IP、端口、健康状态）。
2. Registry 通过心跳（如每 30 秒）判断实例是否存活，失效则剔除。
3. Consumer 启动时拉取并缓存实例列表，定期刷新。
4. Consumer 调用时，借助负载均衡从列表选一个实例发起请求。

## 三、CAP 与注册中心选型

分布式系统的 CAP 定理告诉我们，一致性（Consistency）、可用性（Availability）、分区容错性（Partition tolerance）三者只能同时满足两个。**网络分区（P）在分布式下不可避免**，所以实际是在 C 和 A 之间取舍：

**CP 型注册中心**（强一致，牺牲部分可用性）：如 ZooKeeper、etcd。主节点宕机时选举期间可能短暂不可用，但保证注册信息一致。适合对一致性要求极高的场景。

**AP 型注册中心**（高可用，最终一致）：如 Eureka、Nacos（AP 模式）。节点间数据允许短暂不一致，但任何节点都能响应查询，可用性更强。微服务场景下更常用，因为“临时拿不到最新实例列表”通常比“注册中心整体不可用”后果轻。

Nacos 较为特殊，支持 CP 和 AP 模式切换，既能做注册中心也能做配置中心。

## 四、配置中心解决什么问题

没有配置中心时，配置写在每个服务的 `application.yml` 里。这带来痛点：改一个数据库连接要重新打包发布所有服务；不同环境（dev/test/prod）配置易混乱；无法动态生效。

**配置中心**（如 Nacos、Apollo、Spring Cloud Config）的价值：
- **集中管理**：所有配置统一存放，一处修改。
- **环境隔离**：按 profile / namespace 区分环境。
- **动态刷新**：配置变更后，服务不重启即可感知（结合 `@RefreshScope`）。
- **版本与回滚**：配置有历史版本，出错可快速回退。

## 五、服务拆分原则

**单一职责**：一个服务只负责一个内聚的业务域，像“订单服务”“用户服务”“库存服务”，对外只暴露必要的接口。

**围绕 DDD 限界上下文（Bounded Context）拆分**：限界上下文是领域模型的边界，跨上下文通过明确的接口（API / 事件）通信，避免共享数据库导致的强耦合。

**高内聚低耦合**：强相关的功能放一起，服务之间依赖尽量少、通过稳定契约（API 契约 / 版本号）通信。

**数据自治**：每个服务拥有自己的数据库，禁止其他服务直接连它的库，只能通过接口访问，保证数据边界清晰。

## 六、服务间通信方式

**REST / HTTP**：简洁通用，配合 OpenFeign 声明式调用，可读性高，跨语言友好。

**RPC（如 Dubbo、gRPC）**：基于二进制协议，性能更高、序列化更快，适合内部高频调用，但有语言/框架绑定。

下面用 `RestTemplate` 演示 Consumer 调用另一个服务：

```java
@RestController
public class OrderController {

    @Autowired
    private RestTemplate restTemplate;

    @GetMapping("/order/{userId}")
    public String createOrder(@PathVariable Long userId) {
        // 直接通过服务名或域名调用用户服务（生产环境配合注册中心 + 负载均衡）
        String userInfo = restTemplate.getForObject(
            "http://user-service/api/user/" + userId, String.class);
        return "下单成功，用户：" + userInfo;
    }
}

@Configuration
class RestConfig {
    @Bean
    @LoadBalanced   // 开启客户端负载均衡，支持直接用服务名寻址
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}
```

`@LoadBalanced` 让 `RestTemplate` 具备从注册中心按服务名解析实例、并在多个实例间做负载均衡的能力。

## 实际开发中的应用 / 常见问题

**问题 1：服务该拆多细？** 不要过度拆分。先按业务域粗粒度拆分，等某个服务确实成为瓶颈或团队变大再细化，避免“分布式单体”。

**问题 2：注册中心挂了怎么办？** 主流注册中心客户端会缓存实例列表，中心短暂不可用仍能按缓存调用；同时应保证注册中心自身高可用（集群部署）。

**问题 3：跨服务数据一致性？** 尽量避免跨库事务，采用最终一致性：通过消息队列（MQ）发事件、下游消费补偿，或用 Saga 模式编排。强一致场景才考虑分布式事务框架（如 Seata）。

**问题 4：配置中心敏感信息？** 数据库密码等应结合加密（如 Nacos 密文 + KMS），不要明文入库。
